# 批量上传 v1 设计（中考知识库）

## 背景与目标

现有上传功能一次仅处理单个文件，用户需要重复操作。新增“批量上传”能力，支持一次选择/拖拽多个文件，并在前端展示每个文件的上传状态，全部完成后自动刷新知识库目录。

目标：

- 支持文件选择多选（`multiple`）
- 支持一次拖拽多个文件
- 前端并发上传（默认并发数 3），其余排队
- 状态展示（v1）：等待 / 上传中 / 成功（显示分类）/ 失败（显示原因）
- 单个文件失败不影响其它文件
- 全部上传完成后自动刷新目录

非目标（v1 不做）：

- 真实百分比进度条
- 断点续传
- 合并请求一次上传多个文件到后端（v1 复用现有单文件 `/upload`）

## 约束

- 前端部署在 Cloudflare Pages（静态）
- 后端部署在 Hugging Face Spaces（FastAPI）
- 后端接口保持不变：`POST /upload`（单文件；zip 仍按后端逻辑解压处理）

## 交互设计（前端）

上传区块增强：

- `<input type="file" multiple>` 支持一次选择多个文件
- 拖拽多个文件到虚线框，全部加入上传队列
- 新增“批量上传队列”列表：
  - 每行展示：文件名、状态、（成功时）分类学科、（失败时）错误信息
  - 状态流转：等待 → 上传中 → 成功/失败

并发控制：

- 同一时刻最多 3 个上传任务处于“上传中”
- 其余保持“等待”
- 任一任务完成（成功/失败）后，队列自动补位启动下一任务

边界：

- 单次最多加入 50 个文件；超出部分提示并忽略
- 清空知识库后清空队列显示

## 数据流

1. 用户选择/拖拽 N 个文件
2. 前端将文件加入队列并渲染状态
3. 上传调度器按并发上限启动 `fetch(API_BASE + "/upload")` 请求
4. 每个请求完成后更新该行状态
5. 队列全部完成后调用 `loadStats()` 刷新目录

## 错误处理

- 单文件上传失败：将该条标记为失败并展示错误信息，继续处理其它文件
- 后端不可用：队列中所有进行中的任务失败，其余待上传可保留为“等待”（由用户重新触发/重新选择）

## 迭代方向（v2）

- XHR upload progress（百分比进度条）
- 并发数可配置
- 批量取消/重试失败项
